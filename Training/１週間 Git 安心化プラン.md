


（毎日30分〜60分を想定）

Day 1：ブランチ運用の型を頭に入れる（最重要）

まずは“流れ”の理解だけでOK。

扱うのはこの３つだけ：

main（本番）

develop（開発）

feature/xxxx（作業用）

理解ポイントは1つだけです。
「feature → develop → main」この流れ以外は基本しない。
これさえ染み込んでいれば、案件で混乱しません。



Day 2：実際にブランチを切る練習

手を動かして１回やると一気に楽になります。

やること：

git checkout -b feature/test-branch

適当にファイル１つ書き換え

git add . && git commit -m "test commit"

git push origin feature/test-branch

“うまくいった感覚”を掴む日です。

Day 3：Pull とコンフリクト発生の練習

ほんとはこれが一番みんなつまづきます。

練習方法（1人で出来る）：

main ブランチの同じ行を ２つのブランチで違う内容に編集して commit

順番に main に merge してみる

後のほうで必ずコンフリクトが起きる

VSCode のポップアップで解決する練習をする

「コンフリクト＝怖くない」とわかるだけで、実務のプレッシャーが激減します。

Day 4：PRを作る → 修正 → マージ の流れを通しで練習

PR（プルリク）は案件で毎日使うので、ここは重要。

やること：

GitHub で PR を作る

コメントに「レビューお願いします」っぽい文章を書いてみる

追加修正して push → PR に自動で反映される感覚を掴む

最後に自分でマージして閉じる

「PRの粒度」「PRタイトル」「修正の流れ」が理解できます。

Day 5：コミットメッセージの“型”を覚える

センスはいりません。型だけで十分。

おすすめはこれ：

feat: ◯◯機能を追加
fix: バグ修正
refactor: 仕様変更なしの整理
docs: README更新
chore: 雑務（設定変更）


案件では「コミットの意図が読める」だけで評価が高くなります。

Day 6：Git事故対策（実務でよく起きる問題の予習）

ここを知っておくと実務でパニックにならない。

よくある事故３つだけ覚える：

pull し忘れて push → reject される

ファイル消したつもりがないのに消えてる（ブランチ移動ミス）

stash を使ったら戻し方がわからない

この３種類の“ありがち事故”の対処だけ知っておくと、現場では安心感が段違いです。

Day 7：総まとめとして「１人チーム開発のシミュレーション」

最後に通しで1回やります。

main から develop を切る

develop から feature/aaa を切る

作業 → commit → push

PR 作成

develop にマージ

develop → main にリリースする流れを再現

“あ、こうやって１つの作業が世に出るのか”という感覚がつかめます。

◆ この１週間で得られる状態

ブランチ運用が理解できてる

PR が正しく作れる

コンフリクトが怖くない

コミットが読みやすい

Git の“事故”を自分で回避できる
