# RESTとは
Webサービス設計に関する設計思想、原則
「変更・拡張・分担・障害に強い、長く使えるWeb設計になる」のがメリット

# 1. クライアントとサーバーの分離（Client-Server）
- クライアントとサーバーは完全に独立している

具体的には
JS（Vueなど）： Chromeなどのブラウザで画面や操作を担当
Laravel / PHP / DB：AWSなどのサーバーでデータ処理を担当
→　それぞれが HTTP 通信を通じてやり取りするが、お互いの実装に直接依存しない構造

➡片方だけ変更できる（画面をVue→別UIにしても、サーバーはそのまま使える）
➡作業を分担しやすい（フロントとバックエンドを別々に開発できる）


# 2. ステートレス
- セッションなどの状態管理をサーバー側で行わない
- 処理に必要な情報は全てクライアント側からのリクエストに含みサーバーで保持しない
- 認証方式は、トークン方式　[[トークン認証とセッション認証]]

➡サーバーが身軽になる（セッションを持たない）
➡サーバーを増やしやすい（UserAはサーバーAというような縛りがない＝どのサーバーでも対応できる）
➡障害に強い（サーバーが落ちても「状態」が消えない→ 再ログイン地獄が起きにくい）



# 3. 接続性(Connectability)
- APIでやり取りする情報の内部に、別の情報へのリンクを含められる
- RESTの原則に従って設計されたシステム同士であれば、情報を円滑に連携できる

➡「このデータの続きはここだよ」とデータ自身が道案内してくれる

# 4. 統一インターフェース(Uniform Interface)
- 情報操作として、HTTPのプロトコルで定義されている次のようなメソッドの利用
- GET/POST/PUT/DELETE

➡実装・テスト・保守が楽

# 5. アドレス可能性(Addressability)
- 取り扱う全てのリソース(システムで扱うデータ）がURLで表現される
- URLを一目見て何の情報にアクセスしているかわかるメリット
例：食品のカテゴリを指すURLには「category=foods」というパラメーターを含めるといった実装方法など

➡何をしているかがわかりやすい

# 6. オンデマンドのコード(Code-On-Demand)
- プログラムコードをサーバーからダウンロードし、クライアント側で実行する構造
- クライアントの機能を一時的に拡張、カスタマイズすることが可能

（プログラムコードの取得や実行によってクライアントの挙動が変化すると、ネットワーク通信が複雑になり、
プロトコルの可視性が低下するという制約があるので、必ずしもこの原則を盛り込む必要はない）

➡ちょっとした動きの変更・機能追加ならアプリの再インストールやアップデートが不要

# 補足１　セッション
① ログインする

サーバー：「OK、あなた用の番号を作るね」
その番号＝セッションID

② ブラウザに保存
セッションIDが Cookie として入る

③ 次のアクセス
ブラウザ：「この番号の人です」
サーバー：「あ、さっきの人だ」

これが セッション。
